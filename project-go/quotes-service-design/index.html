Service Use Cases

Sequence diagram:

This service would be used for the following use cases:

Case 1: Fetching Quotes for all Insurers

The process works as follows:

  1. The UI sends a request for initiating the quotes process. In turn it

    is returned with the request id from the quotes service.

  2. UI can then open a SSE connection to the service mesh.

  3. On the Quotes service, it saves the request in a collection with the

    request id and the request payload.

  4. It then fetches data from master service about the insurers,

    categories and products available with the add-on information with them.

  5. For each insurer, we create push tasks in the queue.

  6. The workers consume tasks from the queues and based on the type of

    integration.

  7. Each worker computes the key from the payload and makes an external

    API call or an API call to our master and pricing services to compute the quotes.

  8. After having the quotes, the cache is updated against the key

    computed earlier and also saved into the database.

  9. We also send the data to the SSE stream on the service mesh.

Case 2: Verify Quote

Before proceeding with the payment, we re-calculate the quote and make sure that our proposal and quotes are accurate. The steps largely remain the same with a single difference of a flag marking not to use the cache. This “force_check” flag makes sure that the workers skip cache checks and directly fetches/computes the premium.

This would use a separate priority queue, so that they are computed at a faster pace.

Case 3: Get Quote

insurer_name

A GET API to return code based on “track_id” or “quote_id”. In this case, the service would try to extract the information from the cache and return it. In case the record is not available in cache, it would search the database for the requested record.

Case 4: Validation Failure

In case the request doesn’t have the necessary parameters, we would raise validation error and return the result back to the user without inserting anything into the database.

Possible interactions with other services

Possible interactions with other services:

  • Master Service: for fetching metadata for insurers and plans.

  • Pricing Service: In cases where we have the pricing calculations

    with us, the workers would be accessing the Pricing service for fetching the details.

  • Customer Service: for storing and fetching the customer related

    information. Whenever we would be creating a proposal, we would be getting the customer details, which we would be storing in a central customer service.

  • Track Service - To create the track_id and store it

Proposed service architecture description

Block diagram:

The component/service diagram would be as follows:

The workflow works as follows:

  • A request comes to the Quotes Service. After performing basic

    validations, we generate a track_id (timestamp uuid) from the tracking service and return the response containing the actual payload and track_id.

  • We create a background task, which takes the request payload and

    fetches data from the master service about the insurer and product details.

  • After fetching the list of enabled insurers, the payload is added as

    a task in the queue.

  • The workers dequeue the task and start the process for fetching the

    quote for the insurer.

  • Fetch the quotes from insurers/compute at our end.

  • After the quote is received, it is inserted into the database.

  • An SSE is raised to redis and updated to the UI

In case of a get quotes call:

  • It checks from the database and returns the record.

Database

Database Choice: NoSQL - AWS DocumentDB

The data we are expecting for the quotes would vary from one product to another and would also vary depending on the category of the product. Therefore we would need to go with a NoSQL database for storing the quotes related information. We would have some key fields available for all records, the rest can be any valid json.

The database of choice is AWS DocumentDB, which is a managed MongoDB like service. The advantage of going with AWS DocumentDB is that it uses the same driver as MongoDB. The scaling is taken care of by AWS.

For starting we would like to have a 3 instance (1 primary and 2 replicas) cluster for t3.medium instance type.

Models

We would have two different collections in the database:

  1. Request - This would log the request coming to the server with the

    request id being generated for those. It is mainly to be used for keeping track of the requests that we are getting.
    This would have the following schema:

    a. track_id: UUID of the request coming in to the service

    b. created_at: timestamp of the record

    c. All the other fields coming in the request

Index fields: track_id

  1. Quotes - This collection would have the actual quote data for a

    request. It would consist of a track_id, quote_id and the actual quotes data.
    This would have the following schema:

    a. track_id: UUID of the request

    b. quotes_id: UUID for the specific quote request for an insurer.

    c. created_at: timestamp

    d. All the other quote related fields

Index fields: track_id, quote_id

Trade Offs

  • DocumentDB is an AWS managed service. So the heavy lifting of

    setting up and maintaining a database server goes away. It is easy to scale with zero downtimes during scaling.

  • DocumentDB is MongoDB compliant to v4 of the APIs. This should be

    enough for the scope of this service.

  • One caveat of DocumentDB is that we are not able to access it

    outside of VPC. If we want to access it locally, we need to create a tunnel to a machine which has access to it.

  • Another thing to keep in mind - we can replace DocumentDB with

    MongoDB without making any changes to the code.

Scope

Out of scope items:

  1. Raising of events on activities is currently kept out of scope in

    favor of saving time.

  2. To speed up things we can create the timeuuid track_id in this

    service as well. Once we are ready with the track service, we would move the generation of uuid over there.

  3. Multi-tenancy: Right now the system would be supporting a single

    tenant, but would be made configurable enough that we can modify it to be a multi-tenant system with minimal changes.

Security Considerations

Service Authentication Mechanism

The service would rely that the calls made to it are from an API gateway which takes care of Authentication. We would not accept any request from any other origin. Although for beta launch, we may allow the web services to call the APIs, but those would be redirected as soon as we create the API Gateway.

RBAC

For now we do not support RBAC, but the authentication and authorization would take place on the API Gateway layer.

Data encryption requirements

At rest

There is no PII being stored with the Quotes Service, so we do not need to encrypt the data at rest.

In flight

All the requests would be made over HTTPS.

Performance Considerations

Caching

  • The most frequent outgoing API calls would be to Master Service for

    fetching the metadata of the products and plans. This would be mostly static data which we don’t expect to change often. Therefore we can cache this data (in-memory) on service initialisation and use the data from there. We can assume the TTL for cache to be 1 day and observe if we want to increase it.

  • In cases where we explicitly want to reload the cache, we can expose

    an endpoint for resetting this cache.

  • Also, the workers would reach out to Master Service for fetching

    some additional data. This data can also be cached and the process sped up. Each worker would initially search the cache for the requested data, if not present it would reach out to master service and populate the cache from it. We can use the same endpoint for clearing out the cache.

  • We would also record the request and responses through track_id and

    quotes_id in the cache as well. For such cases, the cache TTL would be 1 hour.
    The track_id key would contain a list of quote_ids. The quote_id key would contain the actual cached data.
    track_id1 -> [q_id1, q_id2, q_id3]
    q_id1 -> Actual quote data
    q_id2 -> Actual quote data
    q_id3 -> Actual quote data

Unit Testing

Framework to be used: Pytest

Mocking library: MagicMock

Report generation: pytest-cov

Any other plugins: No

Reporting

We do not expect any reports to be generated by the system. However, we may occasionally do housekeeping and try to understand the data access patterns to further optimize the service.

Hosting and deployment considerations

  • We would go with K8 based deployments.

  • For development purposes, we would simply use docker containers.

  • This service would not be exposed publicly and would only be

    accessible via API Gateway.

QA Automation technology

We would be using Postman and [Newman]{.underline} to run the API test suites.

Other considerations -

  1. All the timestamps to be stored in epoch time, the conversions to

    happen on the backend.

  2. [Low priority] - For handling multi-tenancy, we would be using a

    separate database per tenant. The details of the tenant would be extracted from the headers.
