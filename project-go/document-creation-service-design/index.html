Document Creation Service

Sl No Date Version Author Change description

1 27-July-2022 1.0 Ipinder Singh Initial draft

Service Use Cases

This service would be used for the following use cases:

Case 1: Creating the policy document

This service would create the policy schedule (i.e policy document) for the given transaction. The user would pass the transaction id, through which this service would extract all information while creating the policy document by reaching out to other services.

After collecting the required information from the other service, a url is generated on which the final policy pdf would be generated.

The same URL would be used for downloading the document

This url with the required params are queued to Kafka, and the url is returned back to the caller.

Worker

The worker would pick up the task from the queue and load the template from the database. After substituting the values, the html would need to be converted to pdf and then saved on the mentioned s3 location.

Each document which is received would be stored in a database and the correct state would be mapped against it.

States a document request would go through:

  • Request received - This would be the initial state, when the request

    is received.

  • Validation error - There was some error in validation

  • In progress - when the worker dequeues the task, it would update the

    state as in-progress

  • Completed - When the policy pdf is generated and is copied to s3, it

    would be marked as completed. It is a terminal state

  • Failed - In case there is some error in creating the pdf or

    uploading it, the request would be marked as failed.

The document_request table would manage the state of the request.

Case 2: Downloading the document

The URL generated in the request would redirect to this service which would return the pdf file generated. In case the file is not generated or the document is not available it would return appropriate return code

Possible interactions with other services.

Possible interactions with other services:

  • Transaction Manager: For fetching the ids for quotes, proposal etc

    for the given transaction id.

  • Master Service: for fetching metadata for vehicle type, vehicle

    category, insurers etc.

  • Quotes Service: For fetching quotes data.

  • Customer Service: For fetching customer related information

  • Proposal service: For fetching proposal information

  • Policy Service: For fetching policy details

Proposed service architecture description

Block diagram:

Database

Database Choice: Postgres

The data to be stored is of fixed schema so the database to be used is Postgres.

Trade Offs

  • Using relational databases as the data we are using is structured.

Scope

Out of scope items:

  1. Only policy pdf for a common insurer being done. We would add

    separate templates for insurers later on.

  2. Multi-tenancy: Right now the system would be supporting a single

    tenant

Security Considerations

Service Authentication Mechanism

The service would rely that the calls made to it are from an API gateway which takes care of Authentication. We would not accept any request from any other origin. Although for beta launch, we may allow the web services to call the APIs, but those would be redirected as soon as we create the API Gateway.

RBAC

For now we do not support RBAC, but the authentication and authorization would take place on the API Gateway layer.

Data encryption requirements

At rest

The s3 bucket would be encrypted.

In flight

All the requests would be made over HTTPS.

Performance Considerations

Caching

  • The most frequent outgoing API calls would be to Master Service for

    fetching the metadata of the products and plans. This would be mostly static data which we don’t expect to change often. Therefore we can cache this data (in-memory) on service initialisation and use the data from there. We can assume the TTL for cache to be 1 day and observe if we want to increase it.

  • In cases where we explicitly want to reload the cache, we can expose

    an endpoint for resetting this cache.

Unit Testing

Framework to be used: Pytest

Mocking library: MagicMock

Report generation: pytest-cov

Any other plugins: No

Reporting

We do not expect any reports to be generated by the system. However, we may occasionally do housekeeping and try to understand the data access patterns to further optimize the service.

Hosting and deployment considerations

  • We would go with K8 based deployments.

  • For development purposes, we would simply use docker containers.

  • This service would not be exposed publicly and would only be

    accessible via API Gateway.

QA Automation technology

We would be using Postman and [Newman]{.underline} to run the API test suites.

Other considerations -

  1. All the timestamps to be stored in epoch time, the conversions to

    happen on the backend.

  2. [Low priority] - For handling multi-tenancy, we would be using a

    separate database per tenant. The details of the tenant would be extracted from the headers.
