Customer Service

Service Use Cases

This service would be used for the following use cases:

Case 1: Storing the customer data

The customer PII and other related information would be stored in customer service. This would serve as a centralized store for all customer information across different business verticals.

The data would be sent in a POST request and in return, the service would return a customer_id.

For the first phase we would not be keeping any validation in storing data. But later on we can decide to have an identifier, like email or mobile number, for identifying unique customers.

Case 2: Updating the customer data

The stored customer data would be updated through a PATCH request.

Case 3: Retrieving customer data

The customer data would be returned using the customer id generated when storing the customer information.

Case 4: Search customer

The customer can be searched on the basis of email or mobile number. A list of matching customers would be returned from the search api.

Possible interactions with other services.

Possible interactions with other services:

  • Proposal Service - The customer information entered in the proposal

    form would be stored in the customer service and the customer_id would be stored in the proposal data.

  • Policy Service - We would need customer information while generating

    policy.

Proposed service architecture description

Block diagram:

Database

Database Choice: MongoDB

The schema of the data can evolve, so the database of choice would be mongo db / AWS DocumentDB.

Scope

Out of scope items:

  1. Restriction on duplicate customers would be added later

  2. Encryption of PII would be added later

Security Considerations

Service Authentication Mechanism

The service would rely that the calls made to it are from an API gateway which takes care of Authentication. We would not accept any request from any other origin. Although for beta launch, we may allow the web services to call the APIs, but those would be redirected as soon as we create the API Gateway.

RBAC

For now we do not support RBAC, but the authentication and authorization would take place on the API Gateway layer.

Data encryption requirements

At rest

The data needs to be stored in an encrypted format

In flight

All the requests would be made over HTTPS.

Performance Considerations

Caching

  • None. Would be added if needed

Unit Testing

Framework to be used: Pytest

Mocking library: MagicMock

Report generation: pytest-cov

Any other plugins: No

Reporting

We do not expect any reports to be generated by the system. However, we may occasionally do housekeeping and try to understand the data access patterns to further optimize the service.

Hosting and deployment considerations

  • We would go with K8 based deployments.

  • For development purposes, we would simply use docker containers.

  • This service would not be exposed publicly and would only be

    accessible via API Gateway.

QA Automation technology

We would be using Postman and [Newman]{.underline} to run the API test suites.

Other considerations -

  1. All the timestamps to be stored in epoch time, the conversions to

    happen on the backend.

  2. [Low priority] - For handling multi-tenancy, we would be using a

    separate database per tenant. The details of the tenant would be extracted from the headers.
